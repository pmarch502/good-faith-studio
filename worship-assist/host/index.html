Host a Library

Bring-your-own storage for Worship Assist: point the app at your own cloud.

How to point Worship Assist at your own cloud storage. The app speaks the S3 API, so it works with Amazon S3 and any S3-compatible provider (Backblaze B2, Cloudflare R2, Wasabi, DigitalOcean Spaces, MinIO, …). The app code is identical across all of them — only the setup differs, and only in how you create the bucket and the access key.

This document is both the human setup guide and the source for the app's in-app “Host a library” instructions. Keep each provider section self-contained.

0. What the app needs from you

At the end of setup you hand the app six values (these fill an S3Config):

ValueWhat it isExample

1. How the library must be laid out

Upload your songs and shared files in this exact folder structure (under the optional prefix). This mirrors how the app already organizes content, so nothing special is needed — just put files in the right folders:

audio/<Song Name>/<Stem>.mp3               e.g. audio/Reckless Love/Bass.mp3
audio/<Song Name>/metadata.json            (optional — bpm/key/section markers)
audio/<Song Name>/Arrangements/<Name>.json (created by the app; you can leave empty)
audio/<Song Name>/Mutes/<Name>.json        (created by the app)
setlists/<Name>.json                       (created by the app)
mashups/<Name>.json                        (created by the app)
  • You only have to upload audio/. The app creates the other folders as people build arrangements, mute sets, set lists, and mashups.
  • Song name = folder name. Whatever you name the folder is the song title.
  • No metadata.json? The song still plays; it just opens without a beat grid until someone sets its tempo/key in the app. (Songs exported from Worship Assist already include it.)

2. Concepts that apply to every provider

Private bucket + a scoped key. Keep the bucket private. The app authenticates with the access key you create — it does not need public access. (A key is required even just to read, because the app lists the bucket to discover songs, and you don't want your catalog publicly listable.)

Two postures — you choose which key(s) to make:

  • Contributor key — read + add/edit, but cannot delete. Give this one to your team. A leaked contributor link can't wipe the library.
  • Owner key — the above plus delete. Keep this one to yourself; use it if you want to delete/reorganize from inside the app.

You can start with just a contributor key and do rare deletes from the provider's console, or make both. Deletion in the app means deleting arrangements / mute sets / set lists / mashups — never the audio.

Turn on versioning (the safety net). With versioning on, every overwrite and delete is recoverable, so even the owner key can't cause permanent loss. (Amazon S3: opt-in, one switch. Backblaze B2: on by default.)

3. Amazon S3

3.1 Create the bucket

  1. S3 console → Create bucket.
  2. Bucket name: globally unique (e.g. grace-worship-library).
  3. Region: pick one close to your team and note its code (e.g. us-east-2).
  4. Block Public Access: leave ON (fully blocked). The app uses a key.
  5. Create.

3.2 Turn on versioning

Bucket → Properties → Bucket Versioning → Edit → Enable → Save.

3.3 Upload your library

Put your songs under audio/ in the layout from §1. Easiest with the AWS CLI:

aws s3 sync ./my-library s3://grace-worship-library/
# ...or, if you chose a prefix "library":
aws s3 sync ./my-library s3://grace-worship-library/library/

(./my-library should contain the audio/ folder.) The console's drag-and-drop upload works too.

3.4 Create the access keys (IAM)

Make two keys — one for you (host), one to share with the team (user). They map to the “Link type” you pick when minting a link in the app:

  • Host — full access: read, write, delete, and upload songs.
  • User — read everything + add/edit/delete the sidecars (arrangements, mute sets, set lists, mashups) but cannot upload or change songs.

The user key is what actually stops a team member from touching the audio — the app's “User” link type only hides the upload UI; the key is the real guard. So mint the User link from the user key, not the host key.

For each: IAM console → Users → Create user (e.g. worship-host / worship-user, no console sign-in) → Permissions → Attach policies directly → Create inline policy → JSON, paste the matching policy below (replace grace-worship-library with your bucket), finish the user, then open it → Security credentials → Create access key → Application running outside AWS → copy the Access key ID and Secret now (shown once).

Host policy (full access):

{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "ListTheLibrary",
      "Effect": "Allow",
      "Action": "s3:ListBucket",
      "Resource": "arn:aws:s3:::grace-worship-library"
    },
    {
      "Sid": "FullObjectAccess",
      "Effect": "Allow",
      "Action": ["s3:GetObject", "s3:PutObject", "s3:DeleteObject"],
      "Resource": "arn:aws:s3:::grace-worship-library/*"
    }
  ]
}

User policy (read all, write sidecars only — no song upload):

{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "ListTheLibrary",
      "Effect": "Allow",
      "Action": "s3:ListBucket",
      "Resource": "arn:aws:s3:::grace-worship-library"
    },
    {
      "Sid": "ReadEverything",
      "Effect": "Allow",
      "Action": "s3:GetObject",
      "Resource": "arn:aws:s3:::grace-worship-library/*"
    },
    {
      "Sid": "WriteSidecarsOnly",
      "Effect": "Allow",
      "Action": ["s3:PutObject", "s3:DeleteObject"],
      "Resource": [
        "arn:aws:s3:::grace-worship-library/audio/*/Arrangements/*",
        "arn:aws:s3:::grace-worship-library/audio/*/Mutes/*",
        "arn:aws:s3:::grace-worship-library/setlists/*",
        "arn:aws:s3:::grace-worship-library/mashups/*"
      ]
    }
  ]
}

The user can write only the sidecar folders; songs live directly at audio/<Song>/<stem>.mp3 + metadata.json, which WriteSidecarsOnly doesn't cover, so uploads are refused (surfaced as a 403, no crash). If you use a prefix, prepend it to each Resource path (e.g. …/library/audio/*/Arrangements/*). Arrangements and mute sets are nested inside audio/<Song>/ — scope by the Arrangements//Mutes/ segment as above, not a blanket audio/* (that would let users upload songs).

4. Backblaze B2

B2 exposes an S3-compatible API, so the same app works — the differences are the console and that B2 uses application keys with capabilities instead of IAM policies. B2 keeps file versions by default, so there's no versioning switch.

4.1 Create the bucket

  1. B2 console → Buckets → Create a Bucket.
  2. Bucket unique name (e.g. grace-worship-library).
  3. Files in Bucket are: Private.
  4. Create.

4.2 Note the endpoint + region

On the bucket's page, B2 shows an Endpoint like s3.us-west-004.backblazeb2.com.

  • Endpoint = that host.
  • Region = the middle segment of it, e.g. us-west-004.

4.3 Upload your library

The web uploader is fine for a few files; for a real upload use the AWS CLI pointed at B2's endpoint (after configuring it with your key from §4.4), or a tool like Cyberduck / rclone:

aws s3 sync ./my-library s3://grace-worship-library/ \
  --endpoint-url https://s3.us-west-004.backblazeb2.com

4.4 Create the scoped application key

The key ID becomes the app's Access key ID, and the application key becomes the Secret access key (shown once).

  • Owner key (can delete): the B2 console works. App Keys → Add a New Application Key → name it → restrict Allow access to Bucket to your bucket → Type of Access: Read and Write → create.
  • Contributor key (no delete): the console's “Read and Write” includes delete, so use the B2 CLI to omit the deleteFiles capability: 
    b2 create-key --bucket grace-worship-library \
  worship-contributor listBuckets,listFiles,readFiles,writeFiles
    (The owner variant just adds ,deleteFiles.) The command prints two values: the keyID and the applicationKey.

Free-tier note: B2 gives 10 GB storage free — plenty for testing a handful of songs, not the whole library.

5. Other S3-compatible providers

The same app works with any S3-compatible object store. To add one, you need its endpoint host and region, then create a bucket and a key scoped to it (read + list + write, optionally delete). The pattern is always the same; only the console and the endpoint change.

Amazon S3s3.<region>.amazonaws.come.g. us-east-2IAM policy⏳
Backblaze B2s3.<region>.backblazeb2.come.g. us-west-004App key capabilities; versioned by default⏳

(Verified = we've confirmed the app against it. Amazon S3 and Backblaze B2 are the first two targets.)

6. Setup checklist

  • Bucket created, private
  • Versioning on (S3) / confirmed default (B2)
  • Library uploaded under audio/ in the §1 layout
  • Scoped key created (contributor and/or owner)
  • Recorded the six values from §0
  • (After the app's connect flow exists) pasted them in and the song list appears
